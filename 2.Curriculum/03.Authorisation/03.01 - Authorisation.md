# Module 3 – Authorisation

## 3.1 What Is Authorisation

* Definition of authorisation and its distinction from authentication
* Components of an authorisation system: identities, entitlements, resources, and the access control matrix
* Dependency of authorisation on authentication

### 3.1.1 Implicit vs. Explicit Authorisation

* Explicit authorisation: a deliberate, specific grant permitting a subject to perform an action on a resource
* Implicit authorisation: access inferred from context, ownership, role membership, or ambient authority rather than a named grant
* Security implications of implicit authorisation: unintended access, scope creep, and difficulty of audit
* The confused deputy problem
  * A high-privilege component is manipulated into misusing its authority on behalf of a lower-privilege caller
  * Classic example: a privileged program is tricked into overwriting a sensitive file that the caller could not access directly
  * Web example: CSRF exploits the browser as a trusted deputy, leveraging its session credentials on behalf of an attacker
  * Mitigations: capability-based access control; explicit authorisation tokens scoped to a specific action; CSRF protection

## 3.2 Principles of Access Control

* Least privilege
* Deny by default
* Segregation of duties

## 3.3 Access Control Models

### 3.3.1 Access Control List (ACL)

* An ACL is a list attached to a resource that specifies which users, groups, or identities are allowed or denied particular actions
* Common permissions include operations such as read, write, execute, or delete
* ACLs are identity-based: the system checks who the subject is, then consults the resource’s list to decide whether access is permitted
* ACLs are common in filesystems, operating systems, and network devices, but can become difficult to manage at scale or when policies depend heavily on context

### 3.3.2 Role-Based Access Control (RBAC)

* Role assignment and permission sets per role
* Centralised role enforcement; avoid hardcoding role names in application logic
* RBAC limitations for fine-grained or context-dependent decisions

### 3.3.3 Attribute-Based Access Control (ABAC)

* User, resource, action, and environment attributes
* Policy expression and evaluation
* Combining RBAC and ABAC
* Policy complexity and the importance of thorough testing

### 3.3.4 Object Capability (OCAP)

* Capabilities as unforgeable references that carry authority to access a specific resource
* Contrast with identity-based (ACL) models: possession of a capability is sufficient; no separate identity lookup is required
* The Principle of Least Authority (POLA): grant only the capability needed for the task, nothing more
* How OCAP prevents confused deputy attacks by design: a caller can only delegate authority it already holds
* Capability attenuation: restricting a capability before delegating it (e.g., read-only view of a resource)
* Capability delegation: passing a capability to another component without escalating authority
* Practical capability-inspired patterns in web systems
  * Pre-signed URLs scoped to a specific resource and time window
  * Bearer tokens scoped to a single resource or action
  * Password reset links
  * OAuth 2.0 access tokens with fine-grained scopes

## 3.4 Session Management in an Authorisation Context

* Sessions as the ongoing proof of an authentication event and the carrier of the authorised identity

### 3.4.1 Session Timeout and Renewal

* Inactivity timeouts proportionate to application sensitivity
* Maximum absolute session lifetime independent of activity, followed by re-authentication
* Server-side session termination
* Rotate session IDs immediately after authentication; invalidate old tokens immediately
* Periodic rotation of long-lived tokens

### 3.4.2 Token-Based vs. Cookie-Based Sessions

* Where session state is stored: server-side (cookie-based) vs. self-contained token (token-based)
* Authentication flow differences
* Statelessness and scalability trade-offs
* Revocation: instant for server-side sessions; requires short expiry or a token denylist for self-contained tokens

### 3.4.3 Logout and Re-authentication

* Server-side session invalidation at the moment of logout
* Removing all authentication artefacts from the client
* Requiring re-authentication for sensitive actions regardless of session validity
* Session age checks; MFA-aware re-authentication
* Visual indicators of authentication state

### 3.4.4 Session Fixation and Hijacking Defences

* Session fixation: regenerate session ID after login; never accept client-supplied session IDs; invalidate old identifiers
* Session hijacking: HTTPS throughout; high-entropy tokens; short lifetimes and rotation; CSRF protection; XSS defences; anomalous session detection

## 3.5 Self-Contained Tokens

* Self-contained tokens carry their own claims and are signed so that the recipient can verify integrity without a server-side lookup
* JSON Web Token (JWT) is the predominant standard for self-contained tokens in web applications

### 3.5.1 Structure and Claims

* Three Base64URL-encoded sections: header, payload (claims), signature
* Registered claims: `exp` (expiry), `nbf` (not before), `iss` (issuer), `aud` (audience), `sub` (subject)
* Encoding vs. encryption: payload is readable by anyone; use JSON Web Encryption (JWE) when confidentiality is required

### 3.5.2 Signing Algorithms

* Symmetric (HMAC): `HS256` – same secret used to sign and verify; suitable only when issuer and verifier are the same service
* Asymmetric (RSA/EC): `RS256`, `ES256` – private key signs, public key verifies; required when tokens cross service boundaries
* Key length and entropy requirements

### 3.5.3 Common Vulnerabilities

* Algorithm confusion: accepting `alg: none`; RS256-to-HS256 downgrade attack
* Weak or shared HMAC secrets susceptible to offline brute force
* Missing claims validation (`exp`, `nbf`, `iss`, `aud`)
* Token replay in the absence of a revocation mechanism
* Insecure token storage: `localStorage` and `sessionStorage` are accessible to JavaScript

### 3.5.4 Best Practices

* Prefer asymmetric algorithms for tokens consumed by multiple services
* Short expiry combined with refresh token rotation
* Validate all relevant claims on every request
* Use JWKS endpoints for key discovery and rotation
* Store tokens in `HttpOnly` cookies where feasible
* Log token validation failures as security events

## 3.6 Object- and Property-Level Authorisation

### 3.6.1 Object-Level Authorisation (OLA)

* Checking whether the authenticated user has rights over the specific object being requested, not just the correct role
* Insecure Direct Object Reference (IDOR): missing OLA check; part of OWASP Broken Access Control
* Enforcement steps: authenticate → extract object identifier → check ownership or delegated access → enforce least privilege
* Using opaque identifiers (UUIDs) to reduce enumeration; not a substitute for authorisation checks

### 3.6.2 Property-Level Authorisation

* Controlling access to individual fields or properties of an object, beyond controlling access to the object itself
* A user may be permitted to read an order record but not the `costPrice` or `internalNotes` fields
* OWASP API Security Top 10: API3:2023 – Broken Object Property Level Authorization (BOPLA)
* Mass assignment vulnerabilities: blindly binding request body fields to a model without an allow-list
* Enforcement: explicit field-level allow-lists; avoid exposing internal model properties in API responses

### 3.6.3 Function-Level Access Control (FLAC)

* Ensuring users can only invoke application functions their entitlements permit
* Server-side enforcement independent of UI controls; hidden UI elements are not access control

## 3.7 Common Authorisation Mistakes

* Over-provisioning of privileges
* Missing authentication before granting access to sensitive resources
* Reliance on client-side enforcement (hidden UI elements, client-supplied role values)
* Hardcoded role names or permission values in application logic
* Trusting user-supplied identifiers for access decisions

## 3.8 Auditing and Monitoring of Access Decisions

* What to log: access grants for sensitive resources, access denials, privilege changes, anomalous patterns
* Required log fields: timestamp, user identity, resource identifier, action, decision, source
* Tamper-resistant audit trails: append-only stores, real-time log streaming, integrity controls
* Alerting: high denial rates from a single identity; off-hours or unusual-location access; admin endpoint access by non-admin accounts; sudden changes in access patterns

## 3.9 Testing Access Control

1. Test all access control levels independently: UI, API, service layer, domain, and data layer
2. Use multiple user roles: regular, privileged, administrative, and unauthenticated
3. Perform both positive testing (correct entitlements grant access) and negative testing (insufficient entitlements deny access)
4. Test object-level authorisation (IDOR tests) as part of the automated regression suite
5. Test property-level authorisation: verify field access is correctly restricted across roles
6. Verify server-side enforcement by bypassing the UI and calling API endpoints directly
7. Include session and token tests: logout invalidation, token expiry rejection, session ID rotation on authentication
8. Use security testing tools such as ZAP to automate HTTP-level access control checks
9. Trace and log authorisation decisions during testing to make failures actionable
