# Supplementary Reading Material – Authorisation

This reading covers the main ideas a developer should understand about authorisation, access control models, session management, and self-contained tokens.

## Session and Identity Management

HTTP is stateless, so web applications need a session mechanism to preserve identity from one request to the next.
For authorisation, that session is the link between a past authentication event and the access control decisions made afterward.
If an attacker can steal or control a session identifier, they may be able to act as the victim without breaking the login process itself.

Session identifiers should come from a cryptographically strong source of randomness, be long enough to resist guessing, and travel only over HTTPS.
They should be stored in `HttpOnly`, `Secure` cookies with an appropriate `SameSite` attribute, and they should never appear in URLs.
Sessions also require sensible inactivity timeouts and absolute lifetimes that match the sensitivity of the system.
Logout should invalidate the session on the server, not just remove a cookie in the browser.
After successful authentication, the application should issue a new session identifier to prevent session fixation.
The OWASP Session Management Cheat Sheet remains the main reference for these controls: <https://cheatsheetseries.owasp.org/cheatsheets/Session_Management_Cheat_Sheet.html>.

It is also important to understand the difference between cookie-based and token-based sessions.
In a cookie-based design, the real session state lives on the server and the cookie value only points to it.
That makes revocation straightforward because deleting the server-side record is enough.
In a token-based design, the token carries claims and is verified locally by signature, often without checking a central store.
That might improve scalability in some architectures, but revocation becomes harder and usually depends on short lifetimes, refresh token handling, or a denylist.
The right choice depends on the threat model, how quickly you need revocation to take effect, and whether stateless verification is worth the trade-offs.

Weak session management often leads directly to impersonation and privilege abuse.

**Session fixation** happens when an attacker sets up or predicts a session identifier before the victim logs in, then waits for the victim to authenticate using that same identifier.
The standard defence is to issue a fresh, random session identifier immediately after successful login and invalidate the old one: <https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/06-Session_Management_Testing/03-Testing_for_Session_Fixation>.

**Session hijacking** is the broader case where an attacker obtains a valid session identifier through eavesdropping, cross-site scripting, network interception, or accidental disclosure in logs or error messages.
Strong transport security, high-entropy identifiers, limited lifetimes, periodic rotation, and anomaly detection all help reduce this risk.

**Cross-site request forgery (CSRF)** is closely related.
Browsers automatically attach cookies to matching requests, so a malicious site may be able to trigger actions in another application using the victim’s authenticated session.
OWASP’s CSRF Prevention Cheat Sheet explains the main defences, including synchroniser tokens and the `SameSite` cookie attribute: <https://cheatsheetseries.owasp.org/cheatsheets/Cross-Site_Request_Forgery_Prevention_Cheat_Sheet.html>.

For testing guidance, see Chapter 6 of the OWASP Web Security Testing Guide, which covers identifier entropy, cookie attributes, timeout handling, and logout behaviour: <https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/06-Session_Management_Testing/>.

The OWASP Application Security Verification Standard (ASVS), Chapter V3 and V7, turns these ideas into concrete verification requirements for web frontend security and session management: <https://owasp.org/www-project-application-security-verification-standard/>.

## Access Control Models

One traditional model is the **access control list (ACL)**.
Each protected resource carries a list of subjects and the actions each subject may perform.
ACLs are easy to reason about in small systems, but they become harder to manage as the number of users and resources grows.
A change to one user’s permissions may require updates in many places.

**Role-Based Access Control (RBAC)** tries to make this manageable by grouping permissions into roles and assigning users to those roles.
This works well when responsibilities are stable and roles genuinely reflect job functions.
It becomes less effective when access depends on data-specific or contextual facts.
A role alone cannot easily express rules such as allowing a customer to view only their own orders.

**Attribute-Based Access Control (ABAC)** goes further by evaluating policies against attributes of the subject, resource, action, and environment.
That makes it possible to express rules based on ownership, time of day, network location, or other context.
ABAC systems need careful policy design, testing, and governance if they are not to become confusing or inconsistent.
NIST SP 800-162 is a useful reference for the ABAC model and its architecture: <https://doi.org/10.6028/NIST.SP.800-162>.

**Object Capabilities (OCAP)** take a different approach.
Instead of looking up a subject in an access control structure, a capability system gives components unforgeable references that already carry the authority to perform a specific action on a specific resource.
If a component has the capability, it can act.

This lines up closely with the **Principle of Least Authority (POLA)**.
A component should receive only the capabilities it requires and no more.

Capability systems also help avoid the **confused deputy** problem because a caller cannot trick a more privileged component into acting on a resource unless the caller can actually supply the relevant authority.
CSRF and Clickjacking attacks are an example of this issue.
Norm Hardy’s original paper on the confused deputy attack is still worth reading: <https://cap-lore.com/CapTheory/ConfusedDeputy.html>.

Modern web systems often borrow capability-style ideas even if they are not full OCAP systems.
Examples include pre-signed URLs limited to one resource and a short time window, OAuth 2.0 access tokens with narrow scopes, password reset links, and single-use bearer tokens for one action.
Michael Koppmann’s thesis gives a practical discussion of how capability ideas fit web application security: <https://repositum.tuwien.at/handle/20.500.12708/18849>.

For a deeper treatment of common misunderstandings about capability systems, see Miller, Yee, and Shapiro’s *Capability Myths Demolished*: <https://srl.cs.jhu.edu/pubs/SRL2003-02.pdf>.

OWASP’s Proactive Controls guidance on C7, Enforce Access Controls, gives practical advice on deny-by-default design, centralised enforcement, and the need for server-side checks regardless of what the UI shows: <https://owasp.org/www-project-proactive-controls/v3/en/c7-enforce-access-controls>.

The OWASP Authorisation Cheat Sheet is a concise implementation guide: <https://cheatsheetseries.owasp.org/cheatsheets/Authorization_Cheat_Sheet.html>.

ASVS Chapter V8 provides testable requirements for least privilege, deny-by-default behaviour, and access control enforcement: <https://owasp.org/www-project-application-security-verification-standard/>.

## Self-Contained Tokens

A self-contained token includes its own claims about the subject, issuer, and audience, and it is signed so that a recipient can verify integrity without necessarily consulting a server-side session store.

The most common example is the **JSON Web Token (JWT)**.
A JWT has three Base64URL-encoded parts: a header, a payload, and a signature.
The header identifies the token type and signing algorithm.
The payload contains the claims.
The signature protects the first two sections from tampering.
A signed JWT is not encrypted, so anyone who has the token can read the payload.
If the claims need confidentiality, the application needs JSON Web Encryption (JWE), not just a signed JWT.

The registered claims in RFC 7519 matter because they carry security meaning.
`exp` limits how long the token remains valid.
`nbf` stops it from being used too early.
`iss` identifies the issuer.
`aud` limits where the token should be accepted.
If an application fails to validate these claims, it becomes easier to replay tokens or to use a token issued for one service against another service that should not trust it.
RFC 7519 is the core reference: <https://datatracker.ietf.org/doc/html/rfc7519>.

JSON Web Signature (JWS), defined in RFC 7515, describes the signing structures behind JWT integrity verification: <https://datatracker.ietf.org/doc/html/rfc7515>.

JSON Web Encryption (JWE) is defined in RFC 7516: <https://datatracker.ietf.org/doc/html/rfc7516>.

Algorithm handling is one of the areas where JWT implementations go wrong.
Symmetric algorithms such as `HS256` use the same secret for signing and verification, so they make sense only when issuer and verifier are the same service or can securely share that secret.
Asymmetric algorithms such as `RS256` and `ES256` separate signing and verification with private and public keys, which is usually the right fit when tokens cross service boundaries.

Two classic failures show why verifiers must be strict.
In the `alg: none` attack, a forged token claims no signature is needed.
In the RS256-to-HS256 confusion attack, a library that accepts both algorithms may be tricked into treating a public RSA key as an HMAC secret.
The defence is simple in principle.
The verifier must fix the expected algorithm in advance and reject anything else.

Self-contained tokens are also harder to revoke than server-side sessions.
That is why short expiry times and refresh token rotation matter so much.
Where possible, tokens should be stored in `HttpOnly` cookies.
Keeping them in `localStorage` or `sessionStorage` makes theft through cross-site scripting much easier.
Failed token validation should be logged as a security event.

OWASP’s JWT Cheat Sheet covers these issues in more detail, including claims validation, algorithm choice, key management, and storage guidance: <https://cheatsheetseries.owasp.org/cheatsheets/JSON_Web_Token_for_Java_Cheat_Sheet.html>.

ASVS Chapter V9 provides testable requirements for self-contained token security: <https://owasp.org/www-project-application-security-verification-standard/>.

## Object-, Property-, and Function-Level Authorisation

Having a valid session or a recognised role is not enough on its own.
The application still has to decide whether this user may access this specific resource or perform this specific action.

**Object-Level Authorisation (OLA)** is the check that answers that question for a particular object.
Even if a user is allowed to read order records in general, they may not be allowed to read someone else’s order.
The application must verify ownership, delegation, or some other explicit entitlement before it returns the object.

When that check is missing, the result is often an **Insecure Direct Object Reference (IDOR)**.
An attacker changes one object identifier to another and the application serves data without confirming that the new object belongs to the caller.
OWASP classifies this under Broken Access Control, and the OWASP API Security Top 10 lists it as API1:2023, Broken Object Level Authorization.
Opaque identifiers such as UUIDs may make guessing harder, but they are not a substitute for an actual authorisation check: <https://owasp.org/API-Security/editions/2023/en/0xa1-broken-object-level-authorization/>.

The OWASP Web Security Testing Guide section OTG-AUTHZ-004 explains how to test for IDOR in practice: <https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/05-Authorization_Testing/04-Testing_for_Insecure_Direct_Object_References>.

**Property-Level Authorisation** applies the same idea to individual fields.
A user may be allowed to read an order but not the `costPrice` or `internalNotes` fields.
A customer may be allowed to update their shipping address but not their own privilege level or account status.
OWASP API Security Top 10 API3:2023, Broken Object Property Level Authorization, covers this pattern as well as mass assignment, where an API blindly binds incoming fields to an internal model and lets callers set values they should never control: <https://owasp.org/API-Security/editions/2023/en/0xa3-broken-object-property-level-authorization/>.

**Function-Level Access Control (FLAC)** concerns the operations a user may invoke.
Hiding an administrative button in the UI does not secure the server-side function behind it.
Every endpoint still needs its own check.
If the server trusts the UI to do the filtering, a caller can bypass the front end and invoke the endpoint directly.
OWASP API Security Top 10 API5:2023, Broken Function Level Authorization, covers this problem: <https://owasp.org/API-Security/editions/2023/en/0xa5-broken-function-level-authorization/>.

## Recommended Developer Takeaways

By the end of this module, a developer should be able to:

* distinguish authorisation from authentication and explain why both matter at every access decision point
* recognise and prevent session fixation, session hijacking, CSRF, and privilege escalation
* choose an access control model such as RBAC, ABAC, or a capability-inspired design based on the application’s needs
* implement object-level and property-level checks that validate the relationship between the authenticated subject and the specific resource or field
* enforce function-level access control on every server-side endpoint instead of relying on the user interface
* review a JWT implementation for algorithm confusion, missing claims validation, and unsafe storage
* implement session controls such as secure cookie attributes, timeout policies, identifier rotation after login, and server-side logout invalidation
